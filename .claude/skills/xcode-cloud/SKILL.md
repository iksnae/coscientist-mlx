# xcode-cloud

How CoScientist builds on **Xcode Cloud**, the failure modes we already
hit and fixed, and how to read/drive Xcode Cloud **headlessly** via the
App Store Connect (ASC) API with [`asc.py`](asc.py). The goal: never
re-debug a solved Xcode Cloud problem, and never wait on copy-pasted logs.

## Project facts (the setup these fixes assume)

- **`CoScientist.xcodeproj` is gitignored**; `project.yml` is the source of
  truth and the project is generated by **xcodegen**. So Xcode Cloud must
  **regenerate it after clone** — that's what `ci_scripts/ci_post_clone.sh`
  does. Anything Xcode would store *in* the project (schemes, capabilities)
  must be declared in `project.yml`, or it won't exist on the CI side.
- **Signing:** team `G98TZJ75HL` (paid), Automatic signing; iCloud
  container `iCloud.com.iksnae.coscientist` (see the `xcode-cloud` entries
  in `docs/MILESTONE-17-*`). Bundle ids `com.iksnae.coscientist.{app,demo}`.
- **All SwiftPM dependencies are public GitHub HTTPS** (apple/\*,
  ml-explore/\*, huggingface/\* incl. transitive ibireme/yyjson,
  swiftgraphs/Grape).

## Failure modes → fixes (already solved; keep them solved)

| Symptom in the build log | Cause | Fix (where) |
| --- | --- | --- |
| "Grant Access to Your Source Code" lists dependency repos you don't own | **Not a blocker.** Public SPM deps are fetched anonymously; the grant step is for *private* repos only. | Grant only your **own** primary repo; leave deps ungranted and click Next. |
| `ci_post_clone.sh` fails: `brew install …` → `curl: (6) Could not resolve host: ghcr.io` | Xcode Cloud's env can't reach **ghcr.io** (Homebrew's package host). github.com *is* reachable. | Don't use Homebrew. Download the **xcodegen release binary from GitHub** and run in place (`ci_post_clone.sh`). |
| Build won't start / no scheme to select; or CI builds nothing | xcodegen emits **no shared schemes** unless declared; the gitignored project has none on the CI side. | Declare a `scheme:` per app target in `project.yml` → emits `*.xcscheme` into `xcshareddata/xcschemes/`. |
| Archive fails: `Macro "…Macros" from package "…" must be enabled before it can be used` | SwiftPM **macro trust** prompt is interactive; headless builds can't answer it. (We hit it via `MLXHuggingFaceMacros`.) | `defaults write com.apple.dt.Xcode IDESkipMacroFingerprintValidation -bool YES` (+ the `…PackagePluginFingerprintValidatation` key) in `ci_post_clone.sh`, before xcodebuild. |

General rule for this project: **a fix that Xcode would persist in the
`.xcodeproj` must instead live in `project.yml` or a `ci_scripts/` script**,
because the CI project is regenerated from scratch each build.

## ASC API — read/drive Xcode Cloud headlessly

[`asc.py`](asc.py) is a tiny ASC API client (GET/POST; signs an ES256 JWT
from a `.p8`). Xcode Cloud is fully covered by the ASC API, so you can pull
failing-build issues and even trigger builds without the web UI.

### Credentials (env only — never commit the `.p8`)

```sh
export ASC_ISSUER=<issuer-uuid>     # ASC ▸ Users and Access ▸ Integrations (top of page)
export ASC_KEY_ID=<key-id>          # the XXXX in AuthKey_XXXX.p8
export ASC_KEY_PATH=~/Downloads/AuthKey_<key-id>.p8   # optional; this is the default
```

- The **Issuer ID is not inside the `.p8`** — it's a separate UUID from the
  Integrations page. For team keys it is often (not always) the same UUID as
  the team id shown in a build URL (`appstoreconnect.com/teams/<uuid>/…`) —
  try that to save a round-trip, but confirm by a real call.
- Requires `pyjwt` + `cryptography` (both already present in this env).
- The `.p8` is a secret: keep it in `~/Downloads`/Keychain, pass via env,
  **never** add it (or the issuer/key id) to the repo.

### Recipes

```sh
# Find product + workflow
python3 asc.py GET "ciProducts"
python3 asc.py GET "ciProducts/<productId>/workflows"

# Trigger a build on a workflow
cat > /tmp/run.json <<JSON
{"data":{"type":"ciBuildRuns","relationships":{"workflow":{"data":{"type":"ciWorkflows","id":"<workflowId>"}}}}}
JSON
python3 asc.py POST "ciBuildRuns" /tmp/run.json     # returns the new run id + number

# Watch + diagnose a run
python3 asc.py GET "ciBuildRuns/<runId>"            # executionProgress=COMPLETE, completionStatus=SUCCEEDED|FAILED
python3 asc.py GET "ciBuildRuns/<runId>/actions"    # which action failed (Build / Archive / …)
python3 asc.py GET "ciBuildActions/<actionId>/issues"   # the actual error message(s)
```

To poll to completion, loop `ciBuildRuns/<runId>` (≈60s interval; builds
take ~5–8 min) until `executionProgress == COMPLETE`, then dump the failed
actions' issues. Run the poller with `Bash run_in_background: true` so it
re-invokes you on completion rather than blocking.

**Triggering a build spends the user's Xcode Cloud compute (billable).**
Confirm before triggering, and prefer fixing forward over re-triggering in
a tight loop.
